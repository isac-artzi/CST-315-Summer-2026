Topic 2 zooms in on the Factory Floor. We meet the work orders (processes), the clipboards attached to them (PCBs), and the tasks on each order that can be split across helpers (threads). We also formalize how the front desk hands a job to the floor: fork() creates a new clipboard, exec() swaps the work order on that clipboard, and wait() blocks the dispatcher until the order returns.

1Program vs process — the most useful distinction in this course

A program is a static collection of instructions and data sitting on disk — the file ls in /bin/ls. A process is a running instance of a program: code loaded into memory, a stack, a heap, file descriptors, a current state, and a unique process ID (PID). One program can have many processes; close all your browser windows and reopen them, and the same program produces brand-new processes with brand-new PIDs.

The OS tracks every process with a Process Control Block (PCB) — a kernel data structure with everything needed to suspend and resume the process: PID, parent PID (PPID), program counter, register snapshot, memory pointers, open file table, accounting info, signal state, scheduling info. Context switching is, mechanically, saving one PCB and loading another.

2The five-state process lifecycle

Most modern OSes model a process as moving through a small set of states: New (being created), Ready (eligible to run, waiting for a CPU), Running (currently on a CPU), Waiting/Blocked (waiting for I/O or a synchronization event), and Terminated (finished, awaiting cleanup). Transitions are caused by events: admission, scheduling, preemption, I/O completion, exit.

Linux extends this with finer-grained states like S (interruptible sleep), D (uninterruptible sleep), T (stopped), Z (zombie — terminated but not yet reaped by its parent). You will see these in ps output. A zombie process exists because its parent has not yet called wait() to read its exit status — the clipboard is still hanging on the rack.

3Threads — multiple workers on one work order

A thread is a single sequential flow of execution within a process. A process can have many threads. All threads in a process share the same address space, open files, and other resources, but each thread has its own program counter, registers, and stack. That sharing is the point: threads can communicate by reading and writing common memory, which is much cheaper than the inter-process communication needed when separate processes want to coordinate.

Threads are how a single program does many things at once: a web browser uses threads for the UI, network I/O, JavaScript execution, and rendering. The trade-off is that shared memory makes concurrency bugs (race conditions, deadlocks) possible. We will tackle that head-on in Topic 3.

4fork(), exec(), wait() — the three calls your shell lives on

fork() creates a new process by copying the current one. After the call, two processes exist: the parent (which gets the child's PID as the return value) and the child (which gets 0). They share the same code right after the fork; everything that distinguishes them happens because they branch on the return value.

exec() (and its variants execv, execvp, execlp) replaces the current process image with a new program. The PID stays the same; the code, data, heap, and stack are replaced. You almost never call exec alone — you fork first, then have the child exec, leaving the parent free to do other work.

wait() (or waitpid()) blocks the parent until a child terminates and reaps its exit status. Without wait, the child becomes a zombie. With wait, the kernel cleans the child up and tells the parent how it finished.

5Interactive mode and batch mode

The shell you build for Project 2 has two operating modes. Interactive mode prints a prompt, reads one line, parses, forks, executes, waits, and loops. Batch mode opens a file passed on the command line, reads commands until EOF, executes them — and the syllabus requires those commands to run concurrently when separated by semicolons, with the shell waiting until all of them finish before returning to the prompt.

The mechanical difference is just where the input comes from (stdin vs a file handle). The interesting difference is concurrency: a batch line like ls -l; touch file; grep foo file2 requires three forks before the first wait, which means your wait logic must handle multiple outstanding children. That's the data-structure question objective 5 asks about: how do you track them all?

Check for understanding

  1. A process calls fork(). List three things that are identical in parent and child immediately after, and three things that differ.
    Show a sample answer Identical: code, data, heap, open file descriptors (initially), working directory, user ID, signal handlers. Different: PID, PPID, return value of fork (0 in child, child PID in parent), resource accounting counters reset, pending signals/alarms cleared in child.
  2. Why does a typical shell call fork + exec instead of just exec?
    Show a sample answer Because exec replaces the calling process. If the shell called exec("ls") directly, the shell itself would be replaced by ls and the user would never get back to a prompt. Forking first creates a child to do the replacement, leaving the shell intact.
  3. What is a zombie process, and how does it get cleaned up?
    Show a sample answer A zombie is a terminated process whose parent has not yet called wait() to read its exit status. The kernel keeps a minimal record (PID + exit status) so the parent can collect it. When the parent calls wait(), the record is freed. If the parent dies first, the zombie is re-parented to init (PID 1), which reaps it.
  4. In a batch line ls; touch f; grep foo f, your shell needs to launch all three concurrently and only return to the prompt after all complete. Sketch the data structure you'd use to track outstanding children.
    Show a sample answer A small array or list of child PIDs is enough. After forking all three, loop calling waitpid(-1, &status, 0) until every PID in the list has been reaped (or use wait() in a loop and decrement a counter). You may also want to store start time and command string for diagnostics.
