1Embedded systems — same OS principles, harder constraints

An embedded system is a computer dedicated to a specific function, usually with tight constraints on power, memory, and predictability. Smartwatches, pacemakers, thermostats, car engine controllers, drone flight controllers — all embedded. They run real OSes (FreeRTOS, Zephyr, embedded Linux, VxWorks) and use the same concepts you've learned: processes, scheduling, memory management, file systems.

What's different is the budget. A microcontroller might have 256 KB of RAM total. There may be no MMU, so virtual memory doesn't apply. Scheduling decisions must complete within a hard real-time deadline. A page fault that lasts 10 ms is fine on a desktop and unacceptable in a flight controller.

Modern callout. The eBPF subsystem in Linux (and Windows now) lets you safely run small, verified programs in the kernel — used heavily for observability and security. Embedded-class verification (resource-bounded, deadline-safe) is creeping into general-purpose OSes through eBPF and Rust kernel modules.
Factory translation. Imagine a tiny one-station factory in a closet with strict deadlines: the line cannot stop, paperwork must be small enough to fit on a single index card, and the supervisor has to make every decision in milliseconds. Same playbook as the big factory, much less slack.

2Trust boundaries and the input surface

Every security analysis starts with two questions: What is the trust boundary? and What enters from outside it? A trust boundary separates code we wrote and control from data we did not write and do not control. Anywhere data crosses that boundary — user input, network packets, files we did not author, environment variables — we are at risk if we treat the data as if we wrote it.

The OS-level boundaries are well-defined: the system call interface separates user mode from kernel mode; the process boundary separates one program's memory from another's. But within a single program, you must mark your own boundaries: anywhere you call system(), exec*(), or assemble a string that becomes a command, you have a boundary to defend.

Factory translation. The front desk is the trust boundary. Customers (input) hand you tickets. You verify each ticket before passing it inside. If you photocopy a ticket and feed it to the line without checking, a forged ticket is now indistinguishable from a real one.

3Command injection — the forged work order

Command injection happens when user-controlled input is concatenated into a shell command. The attacker supplies input that contains shell metacharacters (;, &&, ||, |, `, $(...), newlines), and the shell treats them as command separators instead of data. The user's rm myFile becomes rm myFile; rm -rf / if the program does naive concatenation.

The Assignment 4 spec is direct: implement a virus that intercepts a command, injects arguments without the user knowing, and executes the modified command. Then implement a scanner that flags files containing this pattern. The virus must use exec*(), not system() — because system() is the easy path, and the lesson is that the attack works even against more "secure" interfaces if you build the argv yourself.

4Detection — patterns and provenance

Detection has two flavors. Pattern matching: look for known-suspicious strings or constructs in files — system(, execve( with hardcoded paths to dangerous binaries, suspicious chained metacharacters in argv-construction code, hardcoded "rm -rf" strings. Fast, easy to fool, useful as a first line. Provenance: track where data came from and follow it to where it gets executed (taint analysis). Slower, harder to fool, requires real tooling.

Assignment 4 asks for pattern matching: scan every file in a directory and warn on suspicious content. The Activity 2 scanner in this topic is a working baseline you'll extend for the assignment.

Two layers of defense
flowchart LR
  S[Source: user input, file content, network] --> V{Validate at boundary?}
  V -- yes --> H[Sanitized data]
  V -- no --> U[Untrusted data]
  H --> EX[exec*() — safe]
  U --> EX2[exec*() — exploited]
  D[Detector: scan files for patterns] --> WARN[Warn user]
Factory translation. Pattern matching is the security guard reading every ticket for known forged phrases. Provenance is the auditor following each ticket from its origin to the final action. Both miss things; both together miss less.

5Modern OS security mechanisms

Detection at the program level is the last line of defense. Modern OSes layer defenses underneath:

Memory protections: non-executable stacks (NX), address space layout randomization (ASLR), stack canaries — make exploits harder even when bugs exist. Mandatory access control: SELinux, AppArmor — policies that constrain what a process can do regardless of who runs it, so a compromised process is bounded. Sandboxing: seccomp-bpf, Linux namespaces, capabilities — strip the kernel surface a process can touch. Verified boot: cryptographic chain from firmware to kernel to user space ensures the OS itself hasn't been tampered with.

The takeaway: security is not "write perfect code." It's defense in depth, every layer assuming the layer above failed.

Modern callout. Confidential computing — Intel SGX, AMD SEV, AWS Nitro Enclaves — moves the trust boundary inside the CPU itself, so even a compromised OS cannot read protected memory. Used today for processing health data, cryptographic keys, and high-value workloads in shared cloud infrastructure.
Factory translation. The supervisor checks tickets (program-level validation). The locked office at the back checks IDs before letting anyone in (kernel access control). The whole building has badge readers and cameras (mandatory access control). And the safe inside the office? Only opens for an authorized key (confidential computing).

Check for understanding

  1. A program does sprintf(cmd, "ls %s", user_input); system(cmd);. Sketch a malicious user_input that adds a second command.
    Show a sample answer user_input = "docs ; rm -rf /tmp/*". After sprintf, cmd becomes ls docs ; rm -rf /tmp/*. The shell treats the semicolon as a separator and runs both. The fix is to avoid system() entirely and use execvp("ls", argv) with argv = {"ls", "docs", NULL} — no shell interpretation of metacharacters.
  2. The Assignment 4 spec says "without the use of a system function." Why does this constraint exist?
    Show a sample answer Because using execvp instead of system demonstrates that the attack does not require a shell — you can construct an argv with malicious arguments directly, bypassing the naive "don't use system" defense. The lesson: argv construction is itself an attack surface.
  3. Your scanner flags any file containing the substring rm -rf. What's the false-positive rate going to look like in a real codebase?
    Show a sample answer High. Documentation, comments explaining what code does, makefiles, CI scripts — all legitimately contain rm -rf. Better signals: rm -rf assembled from concatenated strings; rm -rf with a target derived from user input; rm -rf / with no qualifier. The arms race never ends.
  4. Why does an embedded system care about real-time scheduling guarantees in ways a desktop doesn't?
    Show a sample answer Desktop misses are visible but rarely catastrophic — a stutter, a dropped frame. Embedded misses can be physical: an airbag fires too late, an insulin pump misdoses, a robot arm collides. Real-time schedulers (SCHED_FIFO/RR, or full RTOSes) provide bounded worst-case latency, not just good average latency.
