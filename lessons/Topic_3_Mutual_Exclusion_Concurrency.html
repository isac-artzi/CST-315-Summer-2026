Open: The floor with the shared stations

1Critical sections and the race

A critical section is a stretch of code that touches shared state in a way that must not interleave with other threads doing the same. The classic example: two threads each running balance = balance + 1 on a shared variable. The operation is three machine instructions (load, add, store). If both threads load the same value before either stores, one increment is lost. That is a race condition.

Three requirements for any correct mutual exclusion solution: mutual exclusion itself (at most one thread in the critical section), progress (some thread can eventually enter when none is inside), and bounded waiting (no thread waits forever while others repeatedly enter).

Factory translation. Two workers approach the same tool. If both grab it, the tool breaks (or the work order is wrong). Some coordination — a sign, a light, a key — has to prevent simultaneous access. That coordination must be fair: neither worker should be starved.

2Semaphores — counters with a queue

A semaphore is an integer with two atomic operations: wait() (also called P or down) decrements it, blocking if the value would go negative; signal() (also called V or up) increments it and wakes one waiter if any. A binary semaphore (initial value 1) acts as a lock. A counting semaphore (initial value N) controls access to N copies of a resource.

POSIX gives you sem_init, sem_wait, sem_post, sem_destroy in <semaphore.h>. Linux also provides System V semaphores (semget, semop) and lower-level futexes used by pthread mutexes underneath.

3Monitors — the locked tool room

A monitor is a higher-level construct: a module that bundles shared state and the procedures that operate on it, with an implicit lock that ensures only one thread is inside the monitor at a time. Condition variables inside the monitor let a thread wait for a state predicate to hold, releasing the lock while waiting, and signal another thread when the predicate becomes true.

Monitors are easier to reason about because the synchronization is structural — you cannot accidentally enter the critical section without going through the lock. Semaphores require disciplined programmer use (every wait needs a matching signal) and are easy to misuse. Trade-off: monitors are less flexible.

4Deadlock — the four conditions

Deadlock occurs when a set of processes are all waiting for resources held by others in the set. Coffman's four necessary conditions: mutual exclusion (a resource is held by at most one process), hold and wait (a process holding one resource may request more), no preemption (resources are released only voluntarily), and circular wait (a cycle of "X holds resource Y waits for" exists).

Strategies: prevention (negate one of the four conditions by design), avoidance (use information about future requests, e.g. Banker's algorithm), detection and recovery (let it happen, find the cycle, kill or roll back a process), or the syllabus's Assignment 3 approach — a timer that breaks a process out of its wait after a set interval and lets the system try again.

Factory translation. Worker A holds the drill and needs the press. Worker B holds the press and needs the drill. Neither will let go. The line stops. A timer is the floor supervisor walking over after N minutes and saying "drop everything, start over."

5Classical problems you should be able to draw

Three patterns appear repeatedly. Producer/Consumer: one or more producers add to a bounded buffer, one or more consumers remove from it; synchronization tracks empty slots and full slots. Readers/Writers: many readers may share a resource, but a writer needs exclusive access; the design choice is whether to favor readers (potential writer starvation) or writers (potential reader starvation). Dining Philosophers: five philosophers, five forks, each needs two adjacent forks; the naive solution deadlocks symmetrically.

Check for understanding

  1. Why is a binary semaphore not exactly the same as a mutex, even though both can act as a lock?
    Show a sample answer A mutex has ownership: only the thread that locked it can unlock it. A binary semaphore has no owner — any thread can call sem_post. That makes semaphores more flexible for signaling between threads, but also easier to misuse for plain mutual exclusion.
  2. Name Coffman's four necessary conditions for deadlock and pick one that the timer-based strategy in Assignment 3 attacks.
    Show a sample answer Mutual exclusion, hold-and-wait, no preemption, circular wait. The timer attacks no preemption: after the timeout the OS forcibly releases the held resources, letting the system try again.
  3. In Assignment 1 (Producer/Consumer), the spec forbids semaphores and asks you to rely on sleep and wakeup. What is the failure mode if you sleep when the buffer is full and the consumer wakes you up just as you were checking?
    Show a sample answer Lost wakeup: the producer reads "buffer full" then is preempted before going to sleep. The consumer empties the buffer and signals wakeup, but the producer is not yet asleep. The producer then sleeps forever waiting for a wakeup that already happened. This is exactly why semaphores exist.
  4. When would you reach for a monitor instead of a semaphore?
    Show a sample answer When the synchronization is tied to a coherent piece of state that always gets accessed through the same set of operations — a bounded queue, a connection pool, a thread-safe cache. Bundling state and locking into one module reduces the chance of accidental access without locking.
