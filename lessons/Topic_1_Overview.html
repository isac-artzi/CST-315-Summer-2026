1The hardware stack the OS stands on

An operating system manages four kinds of physical resources: a processor that executes instructions, main memory (RAM) that holds running programs and data, I/O modules that move data to and from devices, and a system bus that connects them. Stallings calls these the four major structural components of a computer (Ch. 1).

Each component has its own performance characteristics. The CPU is fast and expensive. RAM is fast but volatile and limited. Disks and devices are slow but large and persistent. The OS exists in large part to mediate the speed and capacity mismatches between these layers.

Factory translation. The CPU is the assembly-line station. RAM is the staging rack beside it. The disk is the off-site warehouse. The bus is the conveyor belt connecting everything. The OS is the management running the whole place.

2What the OS actually does

The OS plays three roles simultaneously. As a user-program interface, it exposes a way to launch and control programs — typically through a shell, a GUI, or an API. As a resource manager, it decides which program gets the CPU, the memory, the disk, and the network, and when. As an abstraction layer, it hides the messy details of specific hardware behind clean concepts like files, processes, and sockets.

A modern OS does this for many programs at once, on hardware with multiple cores, large but limited memory, and a mix of fast and slow devices. The mechanisms that make this work — processes, threads, scheduling, paging, file systems, drivers — are what the rest of this course unpacks.

Factory translation. Management does three things. It takes orders (interface), assigns workers to orders (resource manager), and hides the chaos of the floor from the customer (abstraction). The customer says "run this program"; they don't say "load these bytes into frame 0x4000."

3The shell — your interface to the factory

A shell is a program that reads commands, parses them, and asks the OS to run them. It is the front-desk dispatcher: you hand it a ticket, it figures out which workers need to be involved, and it tells you when the job is done. Common UNIX/Linux shells are bash, zsh, and sh; in this course you will build your own (call it lopeShell).

A shell has two main modes. Interactive mode prompts you for one command at a time. Batch mode reads commands from a file and runs them without prompting. Project 2 will require both.

Behind the scenes, when the shell runs a command, it uses two system calls — fork() to create a child process and exec() to replace that child with the requested program — then wait() to pause until the child finishes. We'll formalize this in Topic 2.

4The UNIX/Linux command surface

UNIX and Linux expose a small, composable set of command-line tools. Each tool does one thing. You combine them with pipes (|), redirection (>, <), and arguments to build larger workflows. Knowing a working subset of these tools is non-negotiable for the rest of this course.

Minimum command surface for this class: ls, cd, pwd, mkdir, rm, mv, cp, cat, less, grep, find, chmod, chown, ps, kill, man, echo, sort, wc, head, tail, and whoami. You should also be comfortable with vi or nano, environment variables, and writing a basic shell script.

Composing tools with pipes
flowchart LR
  cat[cat access.log] --> grep["grep ERROR"]
  grep --> sort["sort"]
  sort --> uniq["uniq -c"]
  uniq --> head["head -10"]
Factory translation. Each command is a single-purpose tool. The pipe is the conveyor belt that hands the output of one tool to the next. You don't build a single giant machine; you chain small ones.

5From source code to a running process

The compile-and-run lifecycle has four stops. Edit with a text editor (vi, nano, VS Code) to produce a .c source file. Compile with gcc to translate it into an executable. Load by invoking the executable from the shell, which causes the OS to create a process and load the program into memory. Execute as the CPU steps through the instructions until the program exits.

Every project in this course follows this loop. Your shell, your scheduler, and your file system manager are all C programs that go through edit → compile → load → execute every time you change them.

Lifecycle of a C program
flowchart LR
  src[hello.c] --> gcc["gcc hello.c -o hello"]
  gcc --> bin[hello executable]
  bin --> shell["./hello"]
  shell --> proc[Process created]
  proc --> ld[Code + data loaded into RAM]
  ld --> run[CPU executes instructions]
  run --> exit[exit status returned to shell]

Check for understanding

  1. Name the three roles an OS plays, and give one concrete example of each from your own laptop.
    Show a sample answer Interface: launching an app from the Dock or by typing in a terminal. Resource manager: when you open Chrome and the OS allocates RAM and CPU time to each tab. Abstraction: opening a "file" without thinking about which disk sectors it lives in.
  2. When you type ls -l at the prompt, what does the shell actually call into the OS to do? Name the system calls.
    Show a sample answer The shell calls fork() to create a child process, exec() (e.g. execvp) in the child to replace its image with ls, and wait() (or waitpid()) in the parent to pause until ls finishes.
  3. What is the difference between a program and a process?
    Show a sample answer A program is a static set of instructions on disk. A process is a running instance of a program, with its own memory, registers, file descriptors, and state. You can have many processes for the same program.
  4. Why does Project 1 ask you to compile and run a HelloWorld in Linux, not just in your IDE?
    Show a sample answer Because every later project requires Linux/UNIX system calls (fork, exec, wait, pipe, etc.) that don't exist or behave differently on Windows. The IDE is editing surface; the program must run in a real UNIX/Linux environment.
