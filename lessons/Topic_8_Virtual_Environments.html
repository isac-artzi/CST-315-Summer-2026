Open: The mini-factories

1Why virtualize

Three reasons. Consolidation: one physical machine runs many logical machines, lifting CPU utilization from typical-bare-metal 10–20% to typical-virtualized 60–80%. Isolation: tenants share hardware without sharing OSes; a kernel panic in one VM doesn't take down the others. Portability: a VM image moves between hosts without reinstalling; a container moves between hosts without reinstalling its dependencies either.

The cloud is built on this. AWS EC2 is hypervisor-managed VMs. Lambda is a fast-start virtualized environment (Firecracker microVMs). ECS and EKS run containers orchestrated by Kubernetes. Every layer trades isolation for density and start time differently.

Factory translation. Instead of building 10 small factories on 10 plots of land, you build one big building and rent out 10 walled-off shop floors. Each tenant thinks they have their own building. The landlord can rebalance space, fix the roof for everyone at once, and turn unused floors back to common space.

2Type-1 vs Type-2 hypervisors

A Type-1 (bare-metal) hypervisor runs directly on the hardware. There is no host OS underneath. Examples: VMware ESXi, Microsoft Hyper-V, Xen, KVM (when configured as the only layer). Best performance and isolation; standard in cloud and data centers.

A Type-2 (hosted) hypervisor runs as an application on a normal host OS. Examples: VMware Workstation, Oracle VirtualBox, Parallels Desktop. Easier to install on a developer laptop; slightly more overhead because guest I/O has to traverse the host kernel.

Factory translation. Type-1 is the building owner sitting in their office, managing tenants directly. Type-2 is a tenant who also sublets — they have their own factory and rent out subdivided spaces inside it. The latter is more convenient for a small landlord but adds an intermediate layer.

3Full virtualization, paravirtualization, and hardware assist

Three approaches to making the guest's privileged instructions safe. Full virtualization traps every privileged instruction with binary translation or hardware-assist; the guest OS is unmodified. Paravirtualization modifies the guest OS to make explicit hypercalls instead of issuing privileged instructions — better performance, but the guest must be ported. Hardware-assisted virtualization uses CPU extensions (Intel VT-x, AMD-V) that let unmodified guests run privileged instructions directly, with the hypervisor only intercepting what truly needs intercepting.

Modern systems combine all three: hardware assist for the common path, paravirtualized device drivers (virtio) for I/O performance, and binary translation only as a fallback for unusual instructions.

Modern callout. Firecracker (used by AWS Lambda and Fargate) is a minimal VMM written in Rust that starts a VM in ~125 ms with ~5 MiB of memory overhead. It strips a normal hypervisor down to only what serverless workloads need — and the trade-off chart is the same one your Assignment 5 table is filling in. Confidential computing extensions (AMD SEV-SNP, Intel TDX) take the next step: the host hypervisor cannot read VM memory at all.

4Containers vs VMs

A container is not a virtual machine. It's a normal process on the host kernel, wrapped in Linux namespaces and cgroups so that it sees its own filesystem, process tree, network stack, and resource limits. There's no guest kernel, no hardware emulation, no boot. Start time: tens of milliseconds. Overhead: nearly zero. Isolation: only as strong as the host kernel.

VMs run a full guest kernel inside hardware-virtualized resources. Start time: seconds. Overhead: hundreds of MB of RAM. Isolation: a guest can crash; the host won't. Choose containers for density and developer ergonomics; choose VMs for hard isolation and multi-tenant workloads where the kernel must be the boundary.

5Picking a virtualization technology — the Assignment 5 question

Assignment 5 asks you to pick a real workload, survey five virtualization products, score them against criteria (performance, cost, disaster recovery, HA, security, scaling, management), and recommend one. The framework: identify the workload's dominant constraint, then map products to that constraint.

Examples of workload-driven choices. Mass-market multi-tenant SaaS hosting: Type-1 hypervisor (ESXi or KVM) for tenant isolation; high investment in management tooling. Internal microservices for a team: containers + Kubernetes for density and CI speed. Serverless event handlers: microVMs (Firecracker) — VM-level isolation with container-level start time. Developer laptop: Type-2 (Docker Desktop, VirtualBox) for ease of install. Embedded multi-OS device: a small hypervisor like Xen with hardware partitioning.

Factory translation. Different tenants need different walls. A shared hallway works for friendly small teams (containers). A locked office with its own utilities works for separate companies (Type-1 VMs). A modular pop-up unit with its own electrical kit works for short-term occupancy (microVMs).

Check for understanding

  1. Why does a Type-2 hypervisor typically have higher I/O overhead than a Type-1?
    Show a sample answer Guest I/O on Type-1 goes guest → hypervisor → hardware. On Type-2 it goes guest → guest driver → hypervisor → host OS → host driver → hardware. The extra layer costs context switches and copies. Paravirtual device drivers (virtio) close most of the gap but not all of it.
  2. A vendor claims their container platform offers "VM-level isolation." What should you ask to evaluate the claim?
    Show a sample answer Do they run each container in its own microVM (e.g., Kata Containers, gVisor with a kernel boundary, Firecracker per pod)? If not, "VM-level isolation" is marketing — the host kernel is still the shared attack surface. Confidential containers add cryptographic hardware boundaries on top, but the underlying isolation primitive is what matters.
  3. Memory overcommit lets a hypervisor expose more total RAM to guests than the host has. How does this work, and where does it go wrong?
    Show a sample answer Guests don't use all their allocated RAM all the time. The hypervisor pages out cold guest memory to host disk (just like Topic 4's VMM does for processes). It goes wrong when every guest suddenly wants all its RAM — the host starts swapping aggressively, latency spikes, and the dreaded "noisy neighbor" effect appears.
  4. Project 7 is a presentation. What's one Topic 8 concept worth showcasing if your shell ran inside a container?
    Show a sample answer Demonstrate that your shell — built around classical OS primitives (fork, exec, semaphores, paging, scheduling, files) — runs inside a Linux namespace/cgroup wrapper that uses the very same kernel primitives. The container is itself an OS-concepts artifact, not magic.
