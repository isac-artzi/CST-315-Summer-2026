1The file abstraction

A file is a named, persistent stream of bytes. That's it. Everything more sophisticated — formats, structure, semantics — is built on top by the program reading or writing the file. The OS guarantees that a file has a name, lives somewhere on disk (or another device), can be opened, read, written, sought, closed, and deleted.

The kernel tracks an open file with a file descriptor: a small integer the process passes to read, write, lseek, and close. Per-process file descriptors point into a system-wide open-file table, which points to inodes. Three file descriptors are opened by default: 0 = stdin, 1 = stdout, 2 = stderr.

Factory translation. A file is a labeled folder in the archive. An inode is the index card describing that folder — where it physically sits, who owns it, when it was last touched. A file descriptor is the checkout slip the worker carries while using the folder.

2Directories and the hierarchical namespace

A directory is itself a file, with contents that map names to inode numbers. The root directory / is the entry point; subdirectories form a tree (or, with symbolic and hard links, a DAG). A pathname like /home/isac/CST-315/Topic2/lopeShell.c is walked one component at a time: lookup home in /, lookup isac in /home, and so on.

Project 6 implements the same idea: a tree of directory nodes, each holding child files and child directories, with create/rename/delete/move/duplicate operations on both. Deleting a non-empty directory requires a recursion (or a flag like -r to confirm intent) so users don't accidentally lose entire subtrees.

Factory translation. Directories are filing cabinet drawers; subdirectories are folders within drawers; files are documents inside the folders. A path is the route from the main hallway down to the specific document.

3File allocation — three classic schemes

The file system has to map a logical file (a stream of bytes) to physical blocks on disk. Three schemes appear in every OS textbook:

Contiguous

Store the file in consecutive blocks. Fast sequential and random reads. External fragmentation. Hard to grow a file past its allocation.

Linked

Each block points to the next. No fragmentation. Sequential reads are fine. Random reads are awful — you have to chase pointers. FAT is a variant with the chain centralized.

Indexed

A per-file index block lists all block addresses. Random access is O(1). Direct/indirect/double-indirect block schemes (Unix inodes) support both small and very large files.

Modern systems (ext4, XFS, Btrfs, ZFS) use extents — runs of contiguous blocks — combined with indexed metadata. The result: good random access without the rigidity of pure contiguous allocation.

Factory translation. Contiguous is "store this whole job's documents on one shelf." Linked is "the first document points to the second, the second to the third." Indexed is "here's a master sheet listing every shelf this job uses."

4I/O — buffering, blocking, and interrupts

Devices are slow compared to CPUs. To bridge the gap, the OS uses buffering — staging data in memory between the device and the program. Reads pre-fetch ahead; writes accumulate before flushing. The kernel's buffer cache (or page cache, since they merged in modern Linux) holds recently-accessed blocks so repeated reads come from RAM.

I/O can be blocking (the process sleeps until the operation completes), non-blocking (the call returns immediately with whatever's available), or asynchronous (the kernel completes the operation in the background and signals when done). Linux's modern io_uring interface (2019+) uses shared submission/completion queues for very high-throughput async I/O — used by every major database engine.

A read system call's journey
flowchart LR
  P[Process: read fd, buf, n] --> SC[syscall trap]
  SC --> K[Kernel: check buffer cache]
  K -- hit --> CP[Copy bytes to user buffer]
  K -- miss --> DRV[Device driver issues read]
  DRV --> WAIT[Process blocked]
  DEV[Device finishes] --> INT[Interrupt]
  INT --> WAKE[Kernel wakes process]
  WAKE --> CP
  CP --> RET[Return to user code]
Factory translation. A truck (I/O device) brings a shipment. While the truck is in transit, the worker who placed the order is parked in the wait queue. The intercom (interrupt) sounds when the truck arrives; the supervisor moves the worker back to the ready queue.

5RAID — when one disk isn't enough

RAID (Redundant Array of Independent Disks) is a family of techniques for combining multiple disks for performance, reliability, or both. The levels you should be able to compare on the quiz:

RAID 0 — striping. Blocks alternate across disks. Doubles throughput on N-disk array. No redundancy — one disk failure loses everything. RAID 1 — mirroring. Every block written to two (or more) disks. Reads can be served from either. Survives any single disk failure. Half the usable capacity. RAID 5 — striping with distributed parity. Parity block per stripe rotates across disks. Survives one disk failure. Read-heavy workloads do well; writes are expensive due to parity computation. RAID 10 — mirror of stripes. Pair up disks into mirrors, then stripe across the mirrors. High performance and reliability; expensive in capacity.

Project 6 doesn't ask you to implement RAID, but you should be able to discuss when each level fits — a database log gets RAID 10, an archive gets RAID 6, a scratch space gets RAID 0.

Factory translation. RAID 0 is "split this load across two trucks for speed." RAID 1 is "send two trucks with the same load in case one breaks down." RAID 5 is "three trucks, the third one carrying a checksum of the first two — losing any one, you can reconstruct."

Check for understanding

  1. A process opens the same file twice with two separate open() calls. Does it get the same file descriptor? The same inode reference?
    Show a sample answer Different file descriptors (each call returns a new one). The same underlying inode, but the two open-file-table entries are independent — each has its own file offset, so reads on fd 3 don't advance fd 4's position.
  2. Linked allocation is bad for random access. Why? Give the worst-case cost in I/O operations to read byte N of a file.
    Show a sample answer To reach block K you must follow K-1 pointers, each requiring its own block read (cache permitting). Worst case is one block-read per file block traversed: O(N) where N is the block count. Indexed allocation reads the index block once, then the target block — O(1) for files that fit in one index block.
  3. Your Project 4 VMM stores swap pages at disk addresses 0x1000–0x5FFF. Your Project 6 file system allocates blocks starting at 0x1000. What happens, and how do you prevent it?
    Show a sample answer They overwrite each other. Files get corrupted with swap data; swap loses pages that processes need to resume. Prevention: a single allocator owns the disk; the VMM and FS each reserve disjoint regions at startup, or the FS treats the swap region as already-allocated. The project guide explicitly warns about this.
  4. Why is the buffer cache so important to performance, given that modern SSDs are themselves fast?
    Show a sample answer Even fast SSDs are at least 100× slower than RAM for random reads. Repeated reads (config files, libraries, indexes) that miss the cache pay the SSD's cost every time. The cache turns repeated reads into RAM accesses. Plus: the syscall overhead alone is significant if cache is bypassed.
