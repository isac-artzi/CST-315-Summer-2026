Memory Management and Virtual Memory

Where this topic sits in the factory

Topic 4 zooms into the Staging Racks and the Off-Site Warehouse. The racks are physical RAM — fast but limited. The warehouse is disk-backed virtual memory — slow but huge. The OS uses a shelf map (page table) so workers can refer to materials by a stable name without knowing where the pieces are actually stored.

When a worker reaches for a piece that isn't on the rack, a runner is dispatched to fetch it from the warehouse. That round trip is a page fault — expensive enough that you build the whole system to avoid it when possible, and graceful when it can't be avoided.

1Why virtual memory exists

Programs are written as if they have a clean, contiguous block of memory from address 0 upward. The reality is messier: many programs share the machine, RAM is finite, and forcing programs to know about each other's memory layout would be a nightmare. Virtual memory solves this by giving each process its own virtual address space, and translating every memory reference into a physical address at access time.

The translation buys three things. First, isolation: process A cannot read process B's memory because their virtual addresses map to different physical locations. Second, flexibility: physical memory does not need to be contiguous; the OS can place a process's pages wherever there's room. Third, and most powerfully, extension: the virtual address space can be larger than physical memory, with the overflow living on disk and brought into RAM only when needed.

Factory translation. Each worker is told "your materials are at shelves 0 through 1000." The shelf map (page table) translates that to real warehouse locations. Workers don't have to share a numbering scheme; they don't have to know whether their shelf 5 is the same as anyone else's; and shelves beyond rack capacity live in the off-site warehouse.

2Paging — fixed-size chunks of memory

Paging divides the virtual address space into fixed-size pages (typically 4 KB) and physical memory into frames of the same size. Each process has a page table that maps each of its virtual pages to a physical frame, or marks the page as "not present" (lives on disk).

A virtual address breaks into two parts: the high bits select a page number; the low bits are an offset within that page. The page table lookup converts the page number to a frame number; the offset passes through unchanged. The result is a physical address.

Factory translation. A page is a fixed-size shelf. A frame is the slot on the rack that holds a shelf. The page table is the shelf map: "shelf #10 is currently on rack slot #3." The offset is which item on that shelf.

3Page tables and the TLB

For every memory reference, the CPU must consult the page table. That's an extra memory access per access — devastating for performance. Hardware mitigates this with a Translation Lookaside Buffer (TLB): a small, fast cache of recent translations. On TLB hit, no extra memory access; on TLB miss, the page table is walked.

Linux uses a multi-level page table (three or four levels depending on architecture) so that the table itself is paged — you don't need to allocate a flat array large enough for the entire 64-bit virtual address space. The downside is more levels of lookup on a miss; the upside is the table footprint scales with usage.

Factory translation. The TLB is the worker's memory — "I just used shelf 10, it's on rack 3, I'll remember that for a few minutes." The multi-level page table is a binder of binders: the master index points to sub-indexes that each cover a region of the warehouse.

4Demand paging and page replacement

Demand paging: a page is brought into RAM only on first access. When a process touches a page that's not present, the CPU raises a page fault; the OS pauses the process, finds a free frame (or evicts one), reads the page from disk, updates the page table, and resumes.

If memory is full, the OS must choose a victim. Replacement algorithms: FIFO (oldest-loaded out — simple but can evict heavily-used pages), LRU (least-recently-used — best theoretical performance but expensive to track exactly), Clock (a practical LRU approximation using a reference bit). Modern Linux uses a variant of clock with multiple lists.

Factory translation. Demand paging is "only fetch from the warehouse when a worker actually asks." When the rack is full and a runner brings in a new shelf, something must come off. FIFO is "kick out whichever shelf has been on the rack the longest." LRU is "kick out the shelf nobody has touched in a while." Clock is "do a quick walk past every shelf looking for ones with no recent use stamp."

5Linux's page management — a modern snapshot

Linux maintains active and inactive lists of pages, scans them periodically, ages reference bits via the clock algorithm, and reclaims inactive pages first under memory pressure. Memory-mapped files (mmap) and the page cache mean that file I/O and virtual memory share the same machinery — when you read a file, its bytes show up in pages of memory that can then be evicted or reused like any other.

Modern features worth knowing about as you build Project 4: copy-on-write (after fork(), parent and child share pages until one writes — then a copy is made), huge pages (2 MB or 1 GB pages reduce TLB pressure for large workloads), and transparent page sharing in virtualized environments (identical pages across VMs are deduplicated).

Factory translation. Linux keeps two queues of shelves: "in active use" and "could be evicted soon." A janitor walks the inactive queue and marks shelves for return to the warehouse. Copy-on-write is the receptionist saying "the new clipboard shares everything with the old one — but the moment either side scribbles, photocopy the shelf and split them."

Check for understanding

  1. A 32-bit virtual address space with 4 KB pages produces page tables of what size, if implemented as a single flat array?
    Show a sample answer 4 KB = 212 bytes, so the offset is 12 bits and the page number is 20 bits. That's 220 = ~1 million entries per process. At 4 bytes per entry, the page table itself is 4 MB. Per process. Hence multi-level page tables.
  2. A process accesses 1000 pages. RAM holds 100 frames. The access pattern visits each page once in order. Which algorithm performs best — FIFO, LRU, or optimal — and by how much?
    Show a sample answer All perform identically: every reference is a fault (the page wasn't there before). 1000 faults regardless. The choice of algorithm matters when there is locality (repeated access to a working set); a one-shot scan has no locality to exploit.
  3. What is "Belady's anomaly" and why is it counterintuitive?
    Show a sample answer For FIFO replacement, adding more frames can sometimes increase the number of page faults. It's counterintuitive because more memory ought to help. The anomaly does not occur for LRU or optimal.
  4. Why does your Project 4 spec say "all pages should be demand paged"? What would be the alternative and its trade-offs?
    Show a sample answer Alternative: pre-paging — load all pages of a segment as soon as the executable starts. Pro: no faults during execution. Con: huge startup latency, wasted memory if many pages are never touched. Demand paging defers work until it's actually needed, at the cost of fault latency on first touch.
